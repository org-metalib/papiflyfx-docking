## Problem

The settings refactoring (phases 1â€“4) introduced `settings.css` which references `-pf-ui-*` CSS token variables
(e.g. `-pf-ui-surface-panel`, `-pf-ui-text-primary`, `-pf-ui-accent`). These tokens are **never injected** as
inline style on any ancestor node, so every CSS rule in `settings.css` resolves to nothing â€” producing invisible
text, missing backgrounds, and broken controls.

Additionally, `SamplesApp` uses **hardcoded dark-only colors** for the top bar, navigation category headings,
content area, and buttons. These do not respond to the light/dark theme toggle and clash with the token-based
styling the settings panel expects.

### Root Cause Analysis

1. **`SettingsPanel`** loads `ui-common.css` and `settings.css` but never calls
   `UiCommonThemeSupport.themeVariables(palette)` or `UiStyleSupport.metricVariables()` on its root.
   Compare with `GitHubToolbar` (line 522â€“529) which correctly builds a root style string from
   `GitHubThemeSupport.themeVariables(...)` + `UiStyleSupport.metricVariables()` + `UiStyleSupport.fontVariables(...)`.

2. **`DefaultDockThemeService.applyTheme()`** only sets `-fx-background-color` on the docking layer.
   It does **not** emit the `-pf-ui-*` CSS variable block, so docked settings panels inside a
   `DockManager` also lack the tokens.

3. **`SamplesApp`** top bar, content area, and navigation renderer use hardcoded hex colors
   (`#3c3c3c`, `#1e1e1e`, `#888888`, `#5b6470`, `#0e639c`, `#bbbbbb`) instead of theme-derived values.
   The `NAVIGATION_THEME` constant is also hardcoded to dark-only colors.

4. **`SettingsToolbar`** uses plain `Button` instances instead of `UiPillButton`; `SettingsSearchBar`
   does not apply `pf-ui-field` or `pf-ui-compact-field` style classes; neither calls
   `UiCommonStyles.ensureLoaded(this)`.
