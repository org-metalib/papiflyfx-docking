# Prompt - Implement Ribbon Side Placement

You are working in `papiflyfx-docking`.

Lead agent: `@core-architect`
Required reviewers: `@ui-ux-designer`, `@qa-engineer`, `@ops-engineer`, `@spec-steward`

## Task

Implement Ribbon 9: allow applications to host the ribbon on the top, bottom, left, or right side of `RibbonDockHost` when that placement is specified.

Use these documents as the source of truth:

- `spec/papiflyfx-docking-docks/2026-04-25-0-ribbon-9/design.md`
- `spec/papiflyfx-docking-docks/2026-04-25-0-ribbon-9/implementation.md`
- `spec/papiflyfx-docking-docks/2026-04-25-0-ribbon-9/progress.md`

## Requirements

1. Add a `RibbonPlacement` model with `TOP`, `BOTTOM`, `LEFT`, and `RIGHT`.
2. Expose placement on `Ribbon` and `RibbonDockHost`.
3. Let applications specify initial ribbon placement from host setup.
4. Default omitted/null placement to `TOP`.
5. Keep existing top-placement behavior compatible.
6. Move the ribbon to the correct `BorderPane` side without replacing the `Ribbon` instance.
7. Make ribbon header, tab strip, group scroller, adaptive layout, and CSS orientation-aware.
8. Preserve command discovery, contextual tabs, QAT state, selected tab, minimized behavior, theme binding, and provider context across placement changes.
9. Persist placement in ribbon session state with compatibility for sessions that do not contain placement.
10. Refactor `SamplesApp` so available samples can be shown and launched through the ribbon interface while reusing `SampleCatalog` / `SampleScene` metadata.

## Constraints

- JavaFX UI remains programmatic; do not introduce FXML.
- Ribbon providers stay placement-agnostic.
- Do not rotate visible text for side placements.
- Keep all new visual styling tokenized through `-pf-ui-*` and `-fx-ribbon-*` variables.
- Keep changes scoped to ribbon placement and samples-app ribbon discovery.
- Preserve existing public behavior unless explicitly changed by the Ribbon 9 docs.

## Suggested Implementation Order

1. Add `RibbonPlacement` and placement properties.
2. Update `RibbonDockHost` placement layout.
3. Add orientation classes and layout behavior in `Ribbon`.
4. Make adaptive layout axis-aware.
5. Update session data and codec.
6. Add/adjust CSS for all placements.
7. Refactor `SamplesApp` to surface samples through ribbon commands.
8. Add tests.
9. Update docs/progress.

## Tests To Add Or Update

1. `RibbonPlacementFxTest`
   - default placement is top;
   - host region changes for top, bottom, left, and right;
   - selected tab and QAT state survive runtime placement changes.
2. `RibbonAdaptiveLayoutFxTest`
   - side placements render non-empty command panels;
   - side-placement menu labels and arrows remain readable;
   - minimized tab activation works in bottom, left, and right placements.
3. `RibbonSessionPersistenceFxTest`
   - placement round-trip;
   - missing placement defaults to top;
   - unknown placement falls back to top without discarding unrelated ribbon state.
4. `SamplesSmokeTest`
   - available samples still enumerate;
   - representative sample ribbon commands are present;
   - launching a sample through the ribbon uses the same sample scene path as catalog navigation.

## Validation Commands

```bash
./mvnw -pl papiflyfx-docking-docks -am -Dtestfx.headless=true test
./mvnw -pl papiflyfx-docking-samples -am -Dtest=SamplesSmokeTest -Dsurefire.failIfNoSpecifiedTests=false -Dtestfx.headless=true test
git diff --check
```

## Definition Of Done

1. Applications can specify top, bottom, left, or right ribbon placement.
2. Existing top placement remains compatible.
3. Placement persists and restores.
4. All four placements are covered by focused FX tests.
5. `SamplesApp` can show and launch available samples through the ribbon interface.
6. Documentation and `progress.md` are updated with implementation and validation results.
