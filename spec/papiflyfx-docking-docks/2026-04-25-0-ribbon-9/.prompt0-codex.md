````text

Task:
Implement Ribbon 9 host-configurable ribbon placement from the design document. Add support for TOP, BOTTOM, LEFT, and RIGHT ribbon placement while preserving existing top-ribbon behavior, provider contracts, command discovery, contextual tab behavior, minimized behavior, Quick Access Toolbar state, adaptive layout, and session compatibility.

Priority:
P2

Lead Agent:
@core-architect

Required reviewers:
@ui-ux-designer for side-ribbon layout, CSS tokens, visual states, and accessibility ergonomics.
@qa-engineer for regression coverage, TestFX/headless validation, persistence compatibility, and sample coverage.
@spec-steward for progress tracking, documentation closure, and definition-of-done review.
@ops-engineer only if Maven, samples build wiring, or broader runtime/demo infrastructure changes are needed.

Impacted modules and paths:
- papiflyfx-docking-docks
- papiflyfx-docking-samples
- spec/papiflyfx-docking-docks/2026-04-25-0-ribbon-9
- Any ribbon documentation or sample README that describes the public placement API

Key invariants:
- Existing applications that do not set placement must continue to render a TOP ribbon.
- Existing saved sessions without placement must restore successfully as TOP.
- Unknown placement strings and malformed non-string placement values must fall back only for placement and must not invalidate minimized state, selected tab id, or Quick Access Toolbar ids.
- RibbonProvider implementations must remain placement-agnostic; do not add side-specific provider SPI in this slice.
- TOP and BOTTOM ribbons remain horizontal; LEFT and RIGHT ribbons use a side-ribbon model with an outside edge tab strip and an inner command content pane.
- LEFT and RIGHT ordering must mirror so the edge tab strip stays on the outside edge of the host.
- Side tab labels must remain readable and non-rotated, with accessible text and tooltips exposing full tab names.
- Minimized side placement keeps the edge tab strip visible and reveals selected-tab commands through a transient overlay flyout without changing the persisted minimized flag.
- Side group sections must support accordion-style collapse without provider-specific metadata.
- Styling must use existing `-pf-ui-*` and `-fx-ribbon-*` token vocabulary.

Acceptance criteria:
1. Add a `RibbonPlacement` enum with `TOP`, `LEFT`, `RIGHT`, and `BOTTOM`.
2. Add `placementProperty()`, `getPlacement()`, and `setPlacement(...)` to both `RibbonDockHost` and `Ribbon`.
3. Keep TOP as the default and preserve current behavior when placement is not explicitly set.
4. Render the ribbon in the correct `BorderPane` region for all four placements while keeping dock content in the center.
5. Render side placements as an outside edge tab strip plus an inner command content pane, with readable tabs, top-positioned side group headers, full-width large side commands, and wrapped smaller controls where appropriate.
6. Make adaptive group sizing axis-aware: width for horizontal placements and height for vertical placements, while preserving collapse-order semantics.
7. Persist optional placement together with minimized state, selected tab id, and QAT ids; restore missing, unknown, and malformed placement values as TOP without dropping other valid state.
8. Add placement and orientation CSS classes:
   - `.pf-ribbon-placement-top`
   - `.pf-ribbon-placement-bottom`
   - `.pf-ribbon-placement-left`
   - `.pf-ribbon-placement-right`
   - `.pf-ribbon-orientation-horizontal`
   - `.pf-ribbon-orientation-vertical`
9. Preserve keyboard focus behavior across QAT, tabs, collapse button, visible commands, side edge strip, overlay flyout, and accordion group headers.
10. Add a deterministic SamplesApp catalog demo comparing TOP and LEFT ribbon hosts using the same representative sample commands.
11. Add focused tests for default compatibility, explicit placements, persistence fallback behavior, CSS class updates, adaptive sizing, minimized activation, and sample catalog registration.
12. Keep `progress.md` current and record validation evidence before closure.

Implementation guidance:
- Start with Phase 1 from `plan.md`: API and session contract. Add tests before or alongside the contract changes so compatibility behavior is pinned down early.
- Continue to Phase 2 only after the host/ribbon placement property and tolerant session persistence behavior are stable.
- For side layout, share command models between horizontal and vertical rendering paths; do not duplicate provider logic or widen provider contracts.
- Treat the side edge tab strip, inner command pane, overlay flyout, and accordion sections as ribbon internals owned by `papiflyfx-docking-docks`.
- Keep UI changes programmatic JavaFX; do not introduce FXML.
- Keep CSS tokenized and scoped to existing ribbon/theme patterns.
- Update `spec/papiflyfx-docking-docks/2026-04-25-0-ribbon-9/progress.md` as each phase completes, including validation commands and remaining risks.
````
