Task:

Implement Ribbon 10 side toolbar placement. Replace the Ribbon 9 `LEFT` and `RIGHT` persistent vertical side-ribbon command pane with a compact IntelliJ-style side toolbar rail plus transient command popovers over dock content.

- Any ribbon documentation or sample README/status docs that describe side placement behavior

Product decisions:

1. Side rail supports both Quick Access Toolbar commands and ribbon tab/action-group entries.
2. Selected side tabs do not need to remain visually selected after popover close.
3. Minimized side placement keeps the compact rail visible but suppresses popover activation.
4. Text-only side rail commands use generated initials or fallback glyphs instead of long visible labels.
5. Do not preserve the Ribbon 9 persistent side command pane behind an opt-in compatibility flag.

Key invariants:

- Existing applications that do not set placement must continue to render a `TOP` ribbon.
- `TOP` and `BOTTOM` must keep the horizontal Ribbon 9 behavior.
- Existing saved sessions without placement must restore successfully as `TOP`.
- Existing saved sessions with `LEFT` or `RIGHT` must restore as side toolbar placements.
- Unknown placement strings and malformed non-string placement values must fall back only for placement and must not invalidate minimized state, selected tab id, or Quick Access Toolbar ids.
- `RibbonProvider` implementations must remain placement-agnostic; do not add side-specific provider SPI.
- The side rail must stay on the outside edge for both `LEFT` and `RIGHT`.
- Labels must remain readable and non-rotated; text-only side rail commands should use fallback glyphs plus full accessible text/tooltips.
- Minimized side placement must not clear the persisted minimized flag and must suppress popover activation.
- Styling must use existing `-pf-ui-*` and `-fx-ribbon-*` token vocabulary.

Acceptance criteria:

1. `RibbonPlacement`, placement properties, and session field names remain unchanged from Ribbon 9.
2. `LEFT` and `RIGHT` render a compact side toolbar rail by default, with no persistent wide command pane.
3. Side rail includes QAT commands and tab/action-group entries in a deterministic order.
4. Activating a non-minimized side rail tab/action entry opens a transient command popover over dock content.
5. Side popover content uses existing tab/group/control data without provider-specific side metadata.
6. Side minimized placement keeps the rail visible but suppresses popover activation for mouse and keyboard paths.
7. Text-only rail entries use generated initials/fallback glyphs and expose full accessible text and tooltips.
8. Side toolbar and popover focus indicators are visible and keyboard traversal remains usable.
9. `LEFT` and `RIGHT` ordering mirrors so the side rail stays on the outside edge.
10. `RibbonPlacementSample` demonstrates the new side toolbar model.
11. Focused tests cover default compatibility, side rail layout, activation, minimized suppression, session fallback behavior, CSS class updates, fallback glyph accessibility, and sample catalog registration/build.
12. `progress.md` is updated with implementation status, validation evidence, remaining risks, and reviewer handoff.

Implementation guidance:

- Preserve the public placement API and session contract; this should be a side-layout refactor, not an API break.
- Keep the horizontal `TOP`/`BOTTOM` path stable and avoid unrelated ribbon refactors.
- Prefer reusing existing command/group models and render paths for popover content.
- Remove or repurpose Ribbon 9 side command pane internals; do not leave a compatibility mode for the old persistent side pane.
- Treat side rail, popover, fallback glyphs, and minimized suppression as ribbon internals owned by `papiflyfx-docking-docks`.
- Keep UI changes programmatic JavaFX. Do not introduce FXML.
- Update tests alongside implementation so compatibility behavior is pinned down early.
- Keep `spec/papiflyfx-docking-docks/2026-04-25-0-ribbon-10/progress.md` current as phases complete.
