## Objective
Implement **Phase 5 — provider migration and test/documentation closure** now that Phases 1-4 are complete and validated on-host. Finish the provider-facing migration from the deprecated raw-node context bridge to typed capabilities, publish contributor guidance for namespaced session extensions, and close the remaining Phase 5 validation/docs gaps without expanding into a broad cleanup sweep.

## Mandatory Scope (Phase 5)

1. **Phase 4 preflight validation (first)**
   - Re-run and record targeted validation before making new Phase 5 changes:
     - `./mvnw -pl papiflyfx-docking-api,papiflyfx-docking-docks -am compile`
     - `./mvnw -pl papiflyfx-docking-api,papiflyfx-docking-docks -am -Dtestfx.headless=true test`
     - `./mvnw -pl papiflyfx-docking-github,papiflyfx-docking-hugo,papiflyfx-docking-samples -am compile`
   - If preflight fails, fix regressions first and record the outcomes in `progress.md` before starting new provider/doc work.

2. **Migrate remaining ribbon providers off `ACTIVE_CONTENT_NODE`**
   - Update the remaining providers that still cast `RibbonContextAttributes.ACTIVE_CONTENT_NODE`:
     - `papiflyfx-docking-github/src/main/java/org/metalib/papifly/fx/github/ribbon/GitHubRibbonProvider.java`
     - `papiflyfx-docking-hugo/src/main/java/org/metalib/papifly/fx/hugo/ribbon/HugoRibbonProvider.java`
   - Providers must resolve executable integrations through `RibbonContext#capability(Class)` using their typed action interfaces (`GitHubRibbonActions`, `HugoRibbonActions`) instead of attribute/node casting.
   - Provider tests must build contexts with typed capabilities and prove command routing/enablement still works when `ACTIVE_CONTENT_NODE` is absent.
   - Keep the transitional bridge in the docks runtime for now unless a small documentation cleanup requires mentioning its temporary status. This prompt is about provider migration, not hard removal of the bridge.

3. **Publish contributor guidance for typed capabilities and namespaced session extensions**
   - Update provider-facing API docs so the intended contract is explicit:
     - `papiflyfx-docking-api/src/main/java/org/metalib/papifly/fx/api/ribbon/RibbonContext.java`
     - `papiflyfx-docking-api/src/main/java/org/metalib/papifly/fx/api/ribbon/RibbonContextAttributes.java`
     - `papiflyfx-docking-api/src/main/java/org/metalib/papifly/fx/api/ribbon/package-info.java`
   - Update dock/runtime docs so future contributors know how to persist extension-owned state through the Phase 4 namespace/codec model:
     - `papiflyfx-docking-docks/src/main/java/org/metalib/papifly/fx/docks/DockSessionStateContributor.java`
     - `papiflyfx-docking-docks/src/main/java/org/metalib/papifly/fx/docks/DockSessionExtensionCodec.java`
     - `papiflyfx-docking-docks/README.md`
   - Document at least the following:
     - `extensions.<namespace>` ownership model and stable namespace expectations
     - contributor/codec responsibilities and malformed-payload isolation behavior
     - stable command/tab identifiers for persisted ribbon/QAT state
     - typed capability usage vs attribute usage (attributes for metadata/visibility, capabilities for action resolution)

4. **Expand provider-facing runtime guidance without changing the runtime model**
   - Add explicit documentation for the Phase 2/3 invariants that downstream providers/hosts must respect:
     - `CommandRegistry` owns canonical command identity
     - `CommandRegistry` is runtime-owned by `RibbonManager` and is expected to be mutated on the FX thread / refresh-driving thread only
     - steady-state refreshes rely on cache reuse and telemetry-backed invariants rather than full node rebuilds
   - Likely touchpoints include:
     - `papiflyfx-docking-docks/src/main/java/org/metalib/papifly/fx/docks/ribbon/CommandRegistry.java`
     - `papiflyfx-docking-docks/README.md`
     - relevant spec notes in `progress.md`
   - Keep this as documentation/hardening work; do not redesign `CommandRegistry` ownership in Phase 5.

5. **Close remaining Phase 5 regression gaps without reopening already-green work**
   - Audit existing Phase 2-4 coverage before adding new tests.
   - Add/extend tests only where Phase 5 still needs proof, especially:
     - provider capability resolution without `ACTIVE_CONTENT_NODE`
     - provider command routing/enablement after the capability migration
     - any compile-critical provider/sample fallout caused by the migration
   - If plan-level items such as extension round-trips or adaptive-layout telemetry are already covered by earlier phases, preserve them and reference that coverage in `progress.md` rather than duplicating tests.
   - Keep tests deterministic under `-Dtestfx.headless=true`.

6. **Phase 5 progress/handoff closure**
   - Update `spec/papiflyfx-docking-docks/2026-04-20-0-ribbon-2/progress.md` with:
     - completed Phase 5 work
     - validation commands + outcomes
     - which docs were updated for contributor onboarding
     - the remaining intentional bridge/non-goal notes (if `ACTIVE_CONTENT_NODE` stays temporarily populated)
     - a handoff block with files changed, reviewer focus, and any residual risks

## Non-Goals

- Do **not** remove the deprecated `RibbonContextAttributes.ACTIVE_CONTENT_NODE` bridge from the docks runtime in this prompt unless the change is trivial and does not broaden scope.
- Do **not** start a broad repository-wide provider cleanup outside the compile-critical Ribbon 2 surfaces.
- Do **not** redesign the ribbon runtime, `CommandRegistry`, or session schema beyond the documentation and migration work above.
- Do **not** rewrite unrelated README/spec material.
