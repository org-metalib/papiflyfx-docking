## Compatibility Policy
Compatibility is intentionally **not required** for Ribbon 2.

## Objective
Implement **Phase 3 — Layout and rendering efficiency** now that Phase 1 (API contracts) and Phase 2 (runtime command architecture / `CommandRegistry` / ID-first QAT) are complete. Leverage the canonical command-identity guarantee from Phase 2 to stop reconstructing JavaFX nodes on every `refresh()` and to make the adaptive collapse path deterministic and instrumentable.

## Mandatory Scope (Phase 3)

1. **Phase 1 + Phase 2 preflight validation (first)**
   - The sandbox could not run Java 25 / Maven Central; both phases still need an on-host green build.
   - Commands to run and record in `progress.md` before new refactors:
     - `./mvnw -pl papiflyfx-docking-api,papiflyfx-docking-docks -am compile`
     - `./mvnw -pl papiflyfx-docking-api,papiflyfx-docking-docks -am -Dtestfx.headless=true test`
     - `./mvnw -pl papiflyfx-docking-samples,papiflyfx-docking-github,papiflyfx-docking-hugo -am compile`
   - If preflight fails, fix Phase 1/2 regressions first and record outcomes in `progress.md` before starting new layout work.

2. **Adaptive layout refactor — reduce control reconstruction**
   - Today `RibbonManager#canonicalizeTabs` rebuilds `RibbonTabSpec`/`RibbonGroupSpec`/`RibbonControlSpec` records on every refresh, and `Ribbon` / `RibbonGroup` / `RibbonControlFactory` then rebuild the JavaFX node tree to match. Now that commands are canonical (stable identity across refreshes), the node tree can be kept across refresh cycles when the spec structure is equivalent.
   - Introduce a rendering cache keyed by `(tabId, groupId, controlId, collapseState)` that reuses existing `RibbonTab` / `RibbonGroup` / control `Node`s when the underlying canonical command is the same. Rebuild only on structural change (control added/removed/reordered, canonical command swapped) or collapse transitions that demand a different visual.
   - Keep collapse semantics from Phase 1 intact: smaller `collapseOrder` values collapse earlier; iterate from the highest-order group downward when width pressure grows; promote back in reverse when width returns.
   - No behavior change from the user's perspective — only fewer `Platform.runLater` batches, fewer `Node` allocations, fewer scene-graph churn events.

3. **Layout instrumentation hooks for deterministic testing**
   - Add a `RibbonLayoutTelemetry` (or similarly named) internal API in `papiflyfx-docking-docks` that reports deterministic events for:
     - `tabRebuild(tabId, reason)`
     - `groupRebuild(groupId, reason)` where `reason ∈ {STRUCTURAL, COLLAPSE, THEME, INITIAL}`
     - `controlRebuild(controlId, reason)`
     - `collapseTransition(groupId, from, to)` with explicit collapse state
     - `nodeCacheHit(kind, id)` / `nodeCacheMiss(kind, id)`
   - Expose a test-visible `TelemetryRecorder` (package-private or `@VisibleForTesting`) that captures the ordered event stream for assertions. Default runtime wiring is a no-op sink — zero allocation when disabled.
   - Back the telemetry channel with Phase 2's canonical-command guarantee: a "same canonical command, same collapse state" observation should always produce `nodeCacheHit`.

4. **Fix pre-existing accumulator double-merge (flagged in Phase 2 progress)**
   - `TabAccumulator(initial)` and `GroupAccumulator(initial)` both call `merge(initial)` in their constructors, and the outer `computeIfAbsent(...).merge(contribution)` merges again. First contribution's groups/controls are counted twice. Remove the in-constructor `merge(initial)` call (or, symmetrically, skip the outer merge when the accumulator was just constructed — prefer the former for clarity).
   - Cover the fix with a targeted `RibbonManagerTest` case that asserts a single tab + single group contribution yields exactly one control in the merged spec.

5. **SVG-capable icon loading path with graceful fallback**
   - Introduce an icon loader in `papiflyfx-docking-docks` that accepts the existing icon URIs/classpath keys and can render SVG resources in addition to raster formats. On SVG parse/render failure, fall back to the current raster path and log at `WARNING` — never surface an exception to the provider caller.
   - Keep the SPI entry point unchanged (`PapiflyCommand.smallIcon` / `largeIcon` remain URI-like strings). Add the SVG capability purely inside the runtime resolver.
   - Choose a minimal SVG rendering dependency (library choice is at agent discretion — document it in the ADR). Avoid pulling in a heavyweight web engine; a lean SVG-to-JavaFX-Image pipeline is acceptable.
   - Document the new icon pipeline in `progress.md` and add a small ADR at `spec/papiflyfx-docking-docks/2026-04-20-0-ribbon-2/adr-0001-svg-icons.md` per `engineering:architecture` guidance.

6. **Phase 3 test closure**
   - Add/extend tests in `papiflyfx-docking-docks` for:
     - node-cache reuse: repeated `refresh()` with identical provider output produces zero `tabRebuild` / `groupRebuild` / `controlRebuild` events (only `nodeCacheHit`).
     - collapse transitions: shrinking the ribbon width triggers `collapseTransition` events in the expected order (highest `collapseOrder` first); growing restores in reverse order; no control loss at any width.
     - telemetry sanity: a structural change (provider adds a control) emits exactly one `controlRebuild` with `STRUCTURAL` reason.
     - accumulator double-merge fix (see item 4).
     - SVG icon fallback: a broken SVG URI renders via raster fallback without throwing.
   - All tests must be deterministic under `-Dtestfx.headless=true`.

## Constraints

- Follow SOLID and module boundaries.
- Preserve ServiceLoader-based provider onboarding.
- Preserve the Phase 2 contract: canonical command identity is stable across refreshes; `CommandRegistry` mutation stays inside `RibbonManager` on the FX thread.
- Keep implementation focus in:
  - `papiflyfx-docking-docks`
  - `papiflyfx-docking-api` (only if minimal API support is unavoidable — e.g., a new `RibbonControlSpec` shape component; prefer runtime-internal changes)
  - `spec/papiflyfx-docking-docks/2026-04-20-0-ribbon-2/**`
- Do not start full provider migration in `papiflyfx-docking-github` / `papiflyfx-docking-hugo` except compile-critical adaptations.
- Do not introduce a new blocking UI thread or scheduling executor; layout work runs on the FX thread as today.
- Do not regress the Phase 2 test surface (`CommandRegistryTest`, `RibbonManagerTest`, `RibbonCommandRegistryFxTest`, `RibbonSessionPersistenceFxTest`). Existing headless tests must continue to pass.
