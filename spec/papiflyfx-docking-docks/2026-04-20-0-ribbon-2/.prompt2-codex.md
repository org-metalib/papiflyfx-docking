## Objective
Implement **Phase 2 — Runtime command architecture** now that Phase 1 contracts are complete.

## Mandatory Scope (Phase 2)

1. **Phase 1 preflight validation (first)**
   - Run the queued validation from `progress.md` before new refactors:
     - `./mvnw -pl papiflyfx-docking-api,papiflyfx-docking-docks -am compile`
     - `./mvnw -pl papiflyfx-docking-api,papiflyfx-docking-docks -am -Dtestfx.headless=true test`
   - If preflight fails, fix Phase 1 regressions first and record outcomes in `progress.md`.

2. **Introduce a runtime `CommandRegistry`**
   - Add stable command ID registration and lookup in `papiflyfx-docking-docks`.
   - Ensure command lifecycle is long-lived and not rebuilt every `getTabs(context)` call.
   - Keep command identity deterministic for the same provider + command ID inputs.

3. **Refactor ribbon materialization path**
   - Update `RibbonManager` so tab/group/control materialization is separated from command-state ownership.
   - Avoid command instance churn caused by repeated contextual tab recomputation.
   - Preserve deterministic ordering and existing provider onboarding via `ServiceLoader`.

4. **Convert QAT runtime state to ID-first**
   - Store/resolve QAT by command IDs, not command object references.
   - Ensure restore works even when contextual tabs are hidden during initial load.
   - Keep behavior robust when IDs are missing or providers are temporarily unavailable.

5. **Phase 2 test closure**
   - Add/extend tests in `papiflyfx-docking-docks` for:
     - command registry identity stability
     - QAT restore with hidden contextual tabs
     - tolerance of missing command IDs on restore
   - Keep tests deterministic for headless execution.
