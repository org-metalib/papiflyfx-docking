## Compatibility Policy
Compatibility is intentionally **not required** for Ribbon 2.
Breaking API/provider/session changes are acceptable when they improve architecture and runtime correctness.

## Objective
Begin Ribbon 2 implementation with Phase 1 as the mandatory scope and Phase 2 bootstrap if time permits.

## Mandatory Scope (Phase 1 — API Contracts)

1. **Remove UI-framework leakage from API command contracts**
   - Refactor ribbon command state contracts in `papiflyfx-docking-api` to be UI-neutral at the SPI boundary.
   - Keep JavaFX adaptation in runtime (`papiflyfx-docking-docks`), not in shared API contracts.

2. **Introduce typed context capability model**
   - Add typed Ribbon context capability access in API/runtime contracts.
   - Deprecate or remove provider reliance on raw `ACTIVE_CONTENT_NODE` casting path.

3. **Clarify contribution semantics**
   - Make reduction/collapse priority semantics explicit and deterministic.
   - Update Javadocs/docs so contributors cannot misread ordering intent.

4. **Document intentional breakage**
   - In code and spec notes, call out Ribbon 2 contract breaks explicitly.

## Optional Scope (Phase 2 bootstrap)

If Phase 1 is stable in the same session, start runtime scaffolding:
- Introduce a `CommandRegistry` concept in `papiflyfx-docking-docks`.
- Move toward QAT ID-first state handling (without requiring full migration completion in this first pass).

## Constraints

- Follow SOLID and module boundaries.
- Preserve ServiceLoader-based provider onboarding.
- Keep changes focused to:
  - `papiflyfx-docking-api`
  - `papiflyfx-docking-docks`
  - `spec/papiflyfx-docking-docks/2026-04-20-0-ribbon-2/**`
- Do not start provider migration (`github`/`hugo`) until Phase 1 contracts are in place.
