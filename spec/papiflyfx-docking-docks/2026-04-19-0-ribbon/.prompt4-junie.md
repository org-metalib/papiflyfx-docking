# Ribbon Implementation Phase 4: Contextual Tabs and Module Adoption

Follow the design specified in `spec/papiflyfx-docking-docks/2026-04-19-0-ribbon/research-gemini.md` and `plan.md` to implement contextual tab resolution and migrate GitHub and Hugo workflows to the Ribbon.

## Lead Agent
`@core-architect`

## Supporting Agents
`@feature-dev` (for GitHub and Hugo command mapping)

## Objective
Implement Phase 4 of the Ribbon Toolbar plan: Add runtime context resolution for contextual tabs in `papiflyfx-docking-docks` and implement `RibbonProvider`s for `papiflyfx-docking-github` and `papiflyfx-docking-hugo`.

## Key Tasks

1.  **Runtime Context Resolution (`papiflyfx-docking-docks`):**
    *   Harden `RibbonManager` and `DockManager` integration to accurately track the active dock/content.
    *   Ensure `RibbonContext` is updated on focus changes across complex docking arrangements.
    *   Implement logic to show/hide contextual tabs based on the active `RibbonContext`.

2.  **GitHub Ribbon Provider (`papiflyfx-docking-github`):**
    *   Implement `RibbonProvider` and register it via `ServiceLoader`.
    *   Map existing GitHub actions (from `GitHubToolbarViewModel`, `GitRepository`, etc.) into a "GitHub" ribbon tab.
    *   Organize commands into groups:
        *   `Sync`: Pull, Push, Fetch.
        *   `Branches`: New Branch, Merge, Rebase.
        *   `Collaborate`: Pull Request, Issues.
        *   `State`: Commit, Stage, Discard.
    *   Use appropriate icons (Octicons) and set `reductionPriority` according to the research doc.

3.  **Hugo Ribbon Provider (`papiflyfx-docking-hugo`):**
    *   Implement `RibbonProvider` and register it via `ServiceLoader`.
    *   Map Hugo CLI commands (from `HugoServerProcessManager`, etc.) into a "Hugo" ribbon tab.
    *   Organize commands into groups:
        *   `Development`: Hugo server (toggle).
        *   `New Content`: Hugo new (split button).
        *   `Build`: Hugo build.
        *   `Modules`: Hugo mod.
        *   `Environment`: Hugo env.
    *   Implement a **contextual** "Hugo Editor" tab that only appears when a Hugo-related dock (e.g., Hugo preview or a Hugo content file in the editor) is active.

4.  **Sample App Integration (`papiflyfx-docking-samples`):**
    *   Verify that the `github` and `hugo` ribbon tabs appear automatically in the sample app due to `ServiceLoader` discovery.
    *   Test the contextual visibility of the Hugo Editor tab by switching between different dock types.

5.  **Testing & Validation:**
    *   Add TestFX tests for contextual tab appearance/disappearance.
    *   Verify that GitHub and Hugo commands correctly trigger their respective logic.
    *   Update `spec/papiflyfx-docking-docks/2026-04-19-0-ribbon/progress.md` after completion.

## Validation
*   Build all affected modules: `./mvnw -pl papiflyfx-docking-docks,papiflyfx-docking-github,papiflyfx-docking-hugo,papiflyfx-docking-samples -am compile`.
*   Run tests: `./mvnw -pl papiflyfx-docking-docks,papiflyfx-docking-github,papiflyfx-docking-hugo -am test`.
*   Manual check: Run the sample app and verify GitHub/Hugo tabs and the contextual Hugo Editor tab behavior.

## Invariants
*   No direct runtime coupling between `github`/`hugo` modules and `papiflyfx-docking-docks` (use SPI).
*   Ribbon commands must wrap existing action logic to avoid duplication.
*   Contextual tabs must be reactive to focus changes.
