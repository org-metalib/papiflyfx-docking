# Ribbon Implementation Phase 1: SPI and Contribution Model

Follow the design specified in `spec/papiflyfx-docking-docks/2026-04-19-0-ribbon/research-gemini.md` and `plan.md` to start the implementation of the Ribbon command surface.

## Lead Agent
`@core-architect`

## Objective
Implement Phase 1 of the Ribbon Toolbar plan: Define the minimal command/ribbon SPI in `papiflyfx-docking-api`.

## Key Tasks

1.  **Define Command Abstraction (`papiflyfx-docking-api`):**
    *   Create a `PapiflyCommand` interface or record.
    *   Attributes: stable command ID, localized label/tooltip, small (16x16) and large (32x32) icon handles (likely as `String` resource paths or `Image` suppliers), enabled/selected properties (`BooleanProperty`), and an execution callback.

2.  **Define Ribbon SPI (`papiflyfx-docking-api`):**
    *   `RibbonProvider`: SPI for modules to contribute tabs.
    *   `RibbonTabSpec`: Descriptor for a ribbon tab.
    *   `RibbonGroupSpec`: Descriptor for a group within a tab, containing controls.
    *   Control descriptors: `RibbonButtonSpec`, `RibbonToggleSpec`, `RibbonSplitButtonSpec`, `RibbonMenuSpec`.
    *   `RibbonContext`: A small abstraction to provide active dock/content state to providers for contextual visibility.

3.  **Package Placement:**
    *   Place these in a new package: `org.metalib.papifly.fx.api.ribbon`.

4.  **ServiceLoader Support:**
    *   Ensure the `RibbonProvider` is ready for `ServiceLoader` discovery.

5.  **Documentation:**
    *   Add Javadoc to all new interfaces and descriptors explaining the contribution model.

## Validation
*   Ensure the new API compiles: `./mvnw -pl papiflyfx-docking-api compile`.
*   Update `spec/papiflyfx-docking-docks/2026-04-19-0-ribbon/progress.md` after completion.

## Invariants
*   Maintain backward compatibility for docking/session restore.
*   Keep commands UI-agnostic (no `javafx.scene.Node` in the API).
*   Follow the established SOLID principles and naming conventions.
