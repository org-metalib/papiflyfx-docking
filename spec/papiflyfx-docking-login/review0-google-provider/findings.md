**Answer**
The module is clearly intended to support Google sign-in: there is a built-in Google provider [GoogleProvider.java](/Users/igor/github/ikatraev/mixhawkmusic-ws/github/papiflyfx-docking/target/papiflyfx-docking/papiflyfx-docking-login-idapi/src/main/java/org/metalib/papifly/fx/login/idapi/providers/GoogleProvider.java#L12), Google-specific settings [AuthenticationCategory.java](/Users/igor/github/ikatraev/mixhawkmusic-ws/github/papiflyfx-docking/target/papiflyfx-docking/papiflyfx-docking-login/src/main/java/org/metalib/papifly/fx/login/settings/AuthenticationCategory.java#L64), and Google endpoint resolution [ProviderSettingsResolver.java](/Users/igor/github/ikatraev/mixhawkmusic-ws/github/papiflyfx-docking/target/papiflyfx-docking/papiflyfx-docking-login/src/main/java/org/metalib/papifly/fx/login/core/ProviderSettingsResolver.java#L60). The flow is system-browser OAuth with a local loopback callback, not direct Gmail password handling [DefaultAuthSessionBroker.java](/Users/igor/github/ikatraev/mixhawkmusic-ws/github/papiflyfx-docking/target/papiflyfx-docking/papiflyfx-docking-login/src/main/java/org/metalib/papifly/fx/login/core/DefaultAuthSessionBroker.java#L303), [DefaultAuthSessionBroker.java](/Users/igor/github/ikatraev/mixhawkmusic-ws/github/papiflyfx-docking/target/papiflyfx-docking/papiflyfx-docking-login/src/main/java/org/metalib/papifly/fx/login/core/DefaultAuthSessionBroker.java#L315).

So the short answer is: conditionally yes, but not as wired by default. If you initialize it like the sample, register `GoogleProvider`, and configure a Google OAuth client ID/secret, a personal `@gmail.com` account should work. If you use the module’s default runtime/factory path, no, it will not perform a real Gmail authentication flow. Also keep `Google Workspace Domain` blank for personal Gmail accounts, because a nonblank value will reject emails outside that domain [DefaultAuthSessionBroker.java](/Users/igor/github/ikatraev/mixhawkmusic-ws/github/papiflyfx-docking/target/papiflyfx-docking/papiflyfx-docking-login/src/main/java/org/metalib/papifly/fx/login/core/DefaultAuthSessionBroker.java#L511).

`./mvnw -pl papiflyfx-docking-login -am test -Dtestfx.headless=true` passes, but the login tests only use fake providers, not a real Google flow [DefaultAuthSessionBrokerTest.java](/Users/igor/github/ikatraev/mixhawkmusic-ws/github/papiflyfx-docking/target/papiflyfx-docking/papiflyfx-docking-login/src/test/java/org/metalib/papifly/fx/login/core/DefaultAuthSessionBrokerTest.java#L45).

that method registers `GoogleProvider` and installs a configured broker in [`SamplesRuntimeSupport`](/Users/igor/github/ikatraev/mixhawkmusic-ws/github/papiflyfx-docking/target/papiflyfx-docking/papiflyfx-docking-samples/src/main/java/org/metalib/papifly/fx/samples/SamplesRuntimeSupport.java#L21), and `LoginSample` uses that shared broker + registry in [`LoginSample`](/Users/igor/github/ikatraev/mixhawkmusic-ws/github/papiflyfx-docking/target/papiflyfx-docking/papiflyfx-docking-samples/src/main/java/org/metalib/papifly/fx/samples/login/LoginSample.java#L34). The settings sample also uses the same runtime in [`SettingsPanelSample`](/Users/igor/github/ikatraev/mixhawkmusic-ws/github/papiflyfx-docking/target/papiflyfx-docking/papiflyfx-docking-samples/src/main/java/org/metalib/papifly/fx/samples/settings/SettingsPanelSample.java#L31), and the Authentication page is discovered through `ServiceLoader` in [`SettingsPanel`](/Users/igor/github/ikatraev/mixhawkmusic-ws/github/papiflyfx-docking/target/papiflyfx-docking/papiflyfx-docking-settings/src/main/java/org/metalib/papifly/fx/settings/ui/SettingsPanel.java#L127) plus the login module’s service file [`META-INF/services/...SettingsCategory`](/Users/igor/github/ikatraev/mixhawkmusic-ws/github/papiflyfx-docking/target/papiflyfx-docking/papiflyfx-docking-login/src/main/resources/META-INF/services/org.metalib.papifly.fx.settings.api.SettingsCategory#L1).

The actual issue in the sample app is lower-level: Google auth is wired, but it is not Google-tuned enough for a durable Gmail session. `GoogleProvider` inherits the generic authorization request builder in [`GoogleProvider`](/Users/igor/github/ikatraev/mixhawkmusic-ws/github/papiflyfx-docking/target/papiflyfx-docking/papiflyfx-docking-login-idapi/src/main/java/org/metalib/papifly/fx/login/idapi/providers/GoogleProvider.java#L10), and that builder only sends the generic OIDC params in [`GenericOidcProvider`](/Users/igor/github/ikatraev/mixhawkmusic-ws/github/papiflyfx-docking/target/papiflyfx-docking/papiflyfx-docking-login-idapi/src/main/java/org/metalib/papifly/fx/login/idapi/providers/GenericOidcProvider.java#L77). It does not request Google offline access, while the broker later depends on a refresh token for forced refresh in [`DefaultAuthSessionBroker`](/Users/igor/github/ikatraev/mixhawkmusic-ws/github/papiflyfx-docking/target/papiflyfx-docking/papiflyfx-docking-login/src/main/java/org/metalib/papifly/fx/login/core/DefaultAuthSessionBroker.java#L145) and persists one only if returned in [`DefaultAuthSessionBroker`](/Users/igor/github/ikatraev/mixhawkmusic-ws/github/papiflyfx-docking/target/papiflyfx-docking/papiflyfx-docking-login/src/main/java/org/metalib/papifly/fx/login/core/DefaultAuthSessionBroker.java#L489). So in `SamplesApp`, a personal Gmail account can likely sign in once if you configure a Google client ID in Settings, but refresh/session longevity is incomplete. Also, leave `Google Workspace Domain` blank for personal Gmail; otherwise the broker will reject non-matching emails.

They either only check that provider buttons render in [`SamplesSmokeTest`](/Users/igor/github/ikatraev/mixhawkmusic-ws/github/papiflyfx-docking/target/papiflyfx-docking/papiflyfx-docking-samples/src/test/java/org/metalib/papifly/fx/samples/SamplesSmokeTest.java#L151), or they replace the runtime with an unconfigured stub broker in [`SamplesSmokeTest`](/Users/igor/github/ikatraev/mixhawkmusic-ws/github/papiflyfx-docking/target/papiflyfx-docking/papiflyfx-docking-samples/src/test/java/org/metalib/papifly/fx/samples/SamplesSmokeTest.java#L171), which fakes sign-in in [`DefaultAuthSessionBroker`](/Users/igor/github/ikatraev/mixhawkmusic-ws/github/papiflyfx-docking/target/papiflyfx-docking/papiflyfx-docking-login/src/main/java/org/metalib/papifly/fx/login/core/DefaultAuthSessionBroker.java#L535). If the problem you saw happens only after restoring a saved login tab, that is a separate bug: restore goes through [`LoginStateAdapter`](/Users/igor/github/ikatraev/mixhawkmusic-ws/github/papiflyfx-docking/target/papiflyfx-docking/papiflyfx-docking-login/src/main/java/org/metalib/papifly/fx/login/docking/LoginStateAdapter.java#L33), which recreates the pane via the no-arg [`LoginFactory`](/Users/igor/github/ikatraev/mixhawkmusic-ws/github/papiflyfx-docking/target/papiflyfx-docking/papiflyfx-docking-login/src/main/java/org/metalib/papifly/fx/login/docking/LoginFactory.java#L22) and drops the sample’s configured registry.
